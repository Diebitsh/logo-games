# Конструкторы игр под каждый тип + адаптив

**Дата:** 2026-05-14
**Статус:** утверждён

## Контекст

На сайте LogoGames есть конструктор, позволяющий специалисту самостоятельно
собрать игру. Сейчас это **один универсальный** `ConstructorComponent`
(4 шага: тип → звук → контент → сохранение), который умеет только формат
«выбор картинки». Плеер `CustomPlayerComponent` тоже играет лишь этот формат.

Существует **8 типов игр** (см. `source.json`, поле `type`). У каждого —
своя структура контента и свой процесс создания. Нужен **отдельный
конструктор под каждый тип** и **отдельный игровой режим** в плеере, плюс
правка адаптивной вёрстки.

### Соответствие нумерации

Спецификация заказчика нумерует направления 1–8 в своём порядке; в коде
`type` games соответствует:

| Спека | `type` | Компонент игры (встроенный) | Направление |
|-------|--------|------------------------------|-------------|
| 1 | 1 | `what-sound` | Различение неречевых звуков |
| 2 | 2 | `phonemic-awareness` | Слова-квазиомонимы |
| 3 | 8 | `word-conversion` (заглушка) | Замена/перестановка звука |
| 4 | 3 | `phoneme-analysis` | Есть ли звук в слове |
| 5 | 4 | `sound-position` | Место звука в слове |
| 6 | 5 | `first-or-last-char` | Первый/последний звук |
| 7 | 7 | `make-word-by-sounds` | Место звука относительно других |
| 8 | 6 | `sound-sequence` (заглушка) | Сборка слова из звуков |

Далее везде используется `type` (значение из `source.json`).

## Цели

1. Экран выбора направления → отдельный компонент-конструктор под каждый из 8 типов.
2. Игровой режим в плеере под каждый из 8 типов — созданные игры реально играются.
3. Адаптивная вёрстка: встроенные 8 игр + новые конструкторы + плеер.

## Вне объёма

- Доработка встроенных заглушек `sound-sequence` и `word-conversion` — в
  `source.json` нет игр типов 6 и 8, эти компоненты остаются заглушками.
- Серверное хранение — игры остаются в IndexedDB браузера.
- Миграция старых сохранённых игр (schema 1) в новый формат — они просто
  помечаются «устаревший формат».

## Архитектура

### Подход

Хост + общие переиспользуемые блоки + тонкие per-type компоненты. Общий
каркас конструктора и плеера живёт в одном месте; каждый из 8 конструкторов —
небольшой файл, описывающий только свои шаги, валидацию и сборку игры.

### Роутинг (`app.routes.ts`)

- `/constructor` → `ConstructorPickerComponent` — экран выбора из 8 направлений.
- `/constructor/:type` → `ConstructorHostComponent` — общий каркас, через
  `@switch` по `:type` рендерит нужный per-type компонент.
- `/my-games` → без изменений (`MyGamesComponent`).
- `/custom-play?id=...` → `CustomPlayerComponent` — становится хостом со
  `@switch` по `type`.
- Старый `ConstructorComponent` удаляется.

### Структура файлов

```
src/app/constructor/
  custom-game.model.ts          # размеченное объединение CustomGame по type
  custom-games.service.ts       # IndexedDB — без изменений
  drop-file.directive.ts        # без изменений
  constructor-picker.component.*        # экран выбора направления
  constructor-host.component.*          # каркас: степпер, прогресс, кнопки, "Готово", save
  custom-player.component.*             # хост плеера: intro/progress/done + @switch
  shared/
    asset-drop.component.*       # drop-зона картинки/аудио + запись с микрофона
    sound-picker.component.*     # сетка групп звуков, single/multi
    sound-pair-picker.component.*# парные группы для типа 2
  types/
    type1-nonspeech.constructor.*       type1-nonspeech.player.*
    type2-quasi-homonym.constructor.*   type2-quasi-homonym.player.*
    type3-sound-presence.constructor.*  type3-sound-presence.player.*
    type4-sound-position.constructor.*  type4-sound-position.player.*
    type5-first-last.constructor.*      type5-first-last.player.*
    type6-assemble-word.constructor.*   type6-assemble-word.player.*
    type7-relative-position.constructor.* type7-relative-position.player.*
    type8-word-change.constructor.*     type8-word-change.player.*
```

### Общие блоки

**`ConstructorHostComponent`** — шапка «Конструктор», степпер, прогресс,
кнопки Назад/Продолжить, экран «Готово!», вызов `CustomGamesService.save`.
Per-type конструктор предоставляет хосту: список шагов (`{id, label}`),
текущую валидность шага, готовый объект `CustomGame` для сохранения. Связь —
через типизированный интерфейс, host рендерит per-type компонент в `@switch`
и подписывается на его выходы (по образцу `game-layout.component`).

**`AssetDropComponent`** — `kind: 'image' | 'audio'`, `hint: string`,
`value?: CustomGameAsset`, выход `changed`. Для `image` — drag-and-drop /
выбор файла (обёртка над `DropFileDirective`). Для `audio` дополнительно —
кнопка «Записать» (MediaRecorder API): запись с микрофона, превью,
перезапись; альтернатива — перетащить/выбрать файл. Запись конвертируется в
data-URL так же, как загруженный файл.

**`SoundPickerComponent`** — сетка групп звуков и мягких пар (из текущего
шага 2 конструктора), режим `single` или `multi`, выход `selectionChange`.

**`SoundPairPickerComponent`** — парные группы для типа 2:
`[д-т] [з-с] [г-к] [ж-ш] [л-р] [в-ф] [б-п]` и их мягкие пары. Новая
константа `SOUND_PAIRS` в модели.

### Модель данных (`custom-game.model.ts`)

`CustomGame` — размеченное объединение по `type`, `schema: 2`. Общие поля:
`id, schema, name, description?, type, createdAt, updatedAt, cover?`.
`CustomGameAsset` (`{ name, mime, data }`, data — data-URL) сохраняется.

Структура контента по типам:

- **type 1** — `levels: { id, image, audio, distractorImage }[]`
  (картинка+аудио правильного, картинка отвлекающего).
- **type 2** — `soundPair: string` + `levels: { id, correctImage,
  correctAudio, incorrectImage }[]`.
- **type 3** — `sound: string` + `levels: { id, image, audio,
  hasSound: boolean }[]` (`hasSound` — ответ Да/Нет).
- **type 4** — `sound: string` + `levels: { id, image, audio,
  position: 'start' | 'middle' | 'end' }[]`.
- **type 5** — `sound: string` + `levels: { id, image, audio,
  mode: 'first' | 'last', correctSoundImage, incorrectSoundImage }[]`.
- **type 6** — `difficulty: 'easy' | 'medium' | 'hard'` + `levels: { id,
  wordAudio, word }[]` (`word` — что собирать; делится на буквы при игре).
- **type 7** — `sounds: string[]` + `levels: { id, image, word,
  letters: string[] }[]` (`letters` — разбор слова по буквам, который
  специалист может поправить; вопросы генерируются на лету при игре).
- **type 8** — `sound: string` + `levels: { id, sourceWord: string,
  insertSound: string, resultImage }[]`.

`CustomGamesService` не меняется (хранит любой JSON по `keyPath: id`).
Старые игры без `schema` или `schema !== 2` в `MyGamesComponent` и плеере
помечаются «устаревший формат»: карточка показывается, кнопка «Играть»
заблокирована.

### Конструкторы — шаги по спецификации

Каждый конструктор работает в каркасе `ConstructorHostComponent`. Последний
шаг всегда — «Сохранение» (название + описание), затем экран «Готово!».
Большинство конструкторов позволяют добавлять несколько заданий
(«Добавить задание»), как сейчас.

1. **type 1 — Неречевые звуки.** Шаг «Контент»: картинка + аудио правильного
   ответа → картинка отвлекающего ответа (без аудио). → Сохранение.
2. **type 2 — Квазиомонимы.** Шаг «Звуки»: выбор парной группы
   (`SoundPairPickerComponent`). Шаг «Контент»: картинка + аудио правильного →
   картинка неправильного (без аудио). → Сохранение.
3. **type 8 — Замена/перестановка звука.** Шаг «Звук»: выбор звука и мягких
   пар (`SoundPicker`, single). Шаг «Контент»: ввод искомого слова + выбор
   вставляемого звука из предложенных + картинка слова-результата.
   → Сохранение.
4. **type 3 — Есть ли звук.** Шаг «Звук»: выбор звука. Шаг «Контент»:
   картинка + аудио. У drop-зоны аудио подсказка: «звук должен звучать так:
   есть ли звук [X] в слове (слово на картинке)?». Отметка Да/Нет
   (`hasSound`). → Сохранение.
5. **type 4 — Место звука.** Шаг «Звук»: выбор звука. Шаг «Контент»:
   картинка + аудио, подсказка «где находится звук [X] в слове…?». Отметка
   правильного ответа: Начало / Середина / Конец. → Сохранение.
6. **type 5 — Первый/последний звук.** Шаг «Звук»: выбор звука. Шаг
   «Контент»: картинка + аудио, подсказка «какой звук в слове … первый
   (или последний)?» с галочкой первый/последний; картинки правильного и
   неправильного звука. → Сохранение.
7. **type 7 — Место относительно других.** Шаг «Звуки»: мультивыбор звуков
   (`SoundPicker`, multi). Шаг «Контент»: картинка + поле ввода слова.
   Система делит слово по буквам и показывает разбор; специалист может
   поправить разбиение перед сохранением. Вопросы генерируются на лету при
   игре. → Сохранение.
8. **type 6 — Сборка слова из звуков.** Шаг «Контент»: запись/загрузка аудио
   слова. У drop-зоны аудио подсказка: «запишите слово, которое нужно будет
   собирать из звуков». Шаг «Сложность»: выбор перемешивания — лёгко
   (меняются два звука) / средне (три) / трудно (все). Слово делится на буквы
   при игре. → Сохранение.

### Плеер — игровые режимы

`CustomPlayerComponent` — хост: экран intro, прогресс, экран «Готово»,
`@switch` по `type` на 8 play-компонентов. Каждый play-компонент получает
типизированную игру, эмитит `answer`/`finish`, использует синглтон
`play/stop` из `common/functions/sounds.functions`.

1. **type 1** — играет аудио → выбор между картинкой правильного и
   отвлекающего ответа.
2. **type 2** — играет аудио слова → выбор между картинкой правильного и
   квазиомонима.
3. **type 8** — показывает искомое слово и картинку-результат → выбор
   вставляемого звука из чипсов (правильный + отвлекающие из группы).
4. **type 3** — картинка + кнопка «Повтор» (аудио-вопрос) → тап Да / Нет.
5. **type 4** — картинка + «Повтор» → тап Начало / Середина / Конец.
6. **type 5** — картинка + «Повтор» → выбор между картинкой правильного и
   неправильного звука.
7. **type 7** — показывает картинку; система генерирует вопрос вида «какой
   звук после/до [X]» из `letters`; внизу варианты ответа (правильный +
   один отвлекающий).
8. **type 6** — кнопка «Повтор» озвучивает слово; на экране слоты под буквы,
   внизу перемешанные плитки букв (порядок перемешивания — по `difficulty`).
   Тап ставит букву в первый свободный слот, повторный тап на плитке в слоте
   возвращает её. Когда все слоты заполнены — проверка: при ошибке надпись
   «Ошибка!» и неверно стоящие буквы улетают назад на исходные места.

### Адаптивная вёрстка

**Встроенные 8 игр** (`src/app/games/*/`):
- Убрать Bootstrap-сетку (`.row` / `.col-6`) и инлайн `width: 180px;
  height: 200px`.
- Сетка вариантов — CSS Grid `repeat(auto-fit, minmax(...))`.
- Картинки — тянущиеся, `width: 100%`, `aspect-ratio`, `object-fit: cover`.
- Тап-цели ≥ 44px, размеры шрифта через `clamp()`.
- `make-word-by-sounds` — бейджи букв в `flex-wrap`.
- Брейкпоинты согласованы с существующими (~640px, ~720px).

**Новые конструкторы и плеер** — адаптивны изначально, на дизайн-токенах из
`styles.scss` (`--tomato`, `--radius`, тени-стикеры и т.д.).

Проверка юзабилити на разрешениях: ~360px, ~414px, ~768px, ~1024px,
desktop — отсутствие горизонтального скролла, читаемость, размер тап-целей.

## Тестирование

- Сборка проходит (`ng build`).
- Ручная проверка: создать игру каждого из 8 типов в конструкторе → она
  появляется в «Моих играх» → открывается в плеере и проходится до конца.
- Запись аудио с микрофона работает в конструкторе (типы 1, 2, 6 и др., где
  есть аудио).
- Старая игра (schema 1) в «Моих играх» помечена «устаревший формат», кнопка
  «Играть» заблокирована, приложение не падает.
- Адаптив: на ~360px нет горизонтального скролла ни в одной из 8 встроенных
  игр, ни в конструкторах, ни в плеере.

## Открытые риски

- Запись с микрофона требует разрешения и HTTPS/localhost; в Capacitor-сборке
  (Android/iOS) — нужны разрешения на микрофон. Если разрешение не выдано —
  fallback на загрузку файла, кнопка записи показывает понятную ошибку.
- `@switch`-хосты (конструктор и плеер) импортируют все per-type компоненты —
  это увеличивает начальный бандл лениво загружаемого роута; при
  необходимости позже можно разнести на отдельные lazy-роуты.
